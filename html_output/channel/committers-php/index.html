@pamil https://github.com/Behat/Gherkin/pull/253

<https://github.com/Behat/Gherkin/pull/253|#253 Compatibility layer for `cucumber/gherkin`>
<p>This adds a compatibility layer that can be used when <code>cucumber/gherkin</code> is also installed, that uses that parser instead and then maps the result back to our 'regular' AST for Behat (and maybe Codeception?) to consume.</p> <p>This means that when parsing edge cases we will conform much more closely with the cucumber core implementations. It also means newer syntax items will be ignored / handled more cleanly rather than triggering odd parsing errors</p> <p>To avoid bumping the minimum PHP version we support, this does not depend directly on <code>cucumber/gherkin</code>; instead there is a static <code>CucumberGherkinLoader::isAvailable()</code> method that can be called to see if the CucumberGherkinLoader can be used instead of the regular one, e.g.</p> <pre><code>if ( class_exists(CucumberGherkinLoader::class) // if supporting older behat/gherkin &amp;amp;&amp;amp; CucumberGherkinLoader::isAvailable() ) { $parser = new CucumberGherkinLoader(...) } else { $parser = new GherkinFileLoader(...) } </code></pre> <p>An alternative option to this would be to merge this into cucumber/gherkin 5.0.0 and let that require cucumber/gherkin (and PHP 8.1), then the version detection can move to Behat - I'm not sure that's warranted yet</p> <p><strong>Motivation</strong></p> <p>Moving to the cucumber parser entirely will reduce the maintenance effort on Behat, and unlock newer features we didn't support yet.</p> <p>That's a big refactor though, and this goes someway towards it by enabling Behat to introduce the cucumber parser partially as an experimental feature (behind a config flag) to detect feature files in the wild that are unparsable, or parsed in unexpected ways.</p> <p><strong>Compatibility</strong></p> <p>• Passes the behat/gherkin etalon tests (tests in PR) with one exception (see Known incompatibilities) • Passes the entire behat/behat test suite (manually tested by me)</p> <p><strong>Mapping notes</strong></p> <p>The majority of the mapping is simple from one tree to another, aside from a few places where the Behat AST doesn't support a feature:</p> <p><strong>Multi-line scenario/background/outline names</strong></p> <pre><code>Scenario: Something awesome Something amazing </code></pre> <p>Cucumber sees this as Scenario with <code>name: 'Something awesome'</code> and <code>description: 'Something amazing'</code>. For compatibility we map this as a multi-line scenario name (which cucumber does not support but Behat does) so <code>title: "Something awesome\nSomething amazing"</code></p> <p><strong>Descriptions trimming</strong></p> <p>Cucumber preserves whitespaces in descriptions, Behat right and left-trims them based on how indented the previous keyword is. This logic is retained to be as like Behat as possible.</p> <p><strong>Rule support</strong></p> <p>Cucumber supports scenarios inside tagged Rules:</p> <pre><code>Feature: @foo Scenario: @bar Rule: @baz Scenario: </code></pre> <p>Because the Behat AST doesn't support Rule, this will be flattened into the same AST as this feature:</p> <pre><code>Feature: @foo Scenario: @bar @baz Scenario: </code></pre> <p>This will enable implementations to parse files with Rule, and to use tags from Rule keywords when filtering examples</p> <p><strong>Known incompatibilities</strong></p> <p>There will doubtless be incompatibilities in the wild, but most will be around 'odd' Gherkin. The following are known:</p> <p><strong>Comments at the end of Feature description no longer work</strong></p> <p><a href="https://github.com/cucumber/common/issues/1413">Cucumber has a bug about this</a> so one of the test files has had to be excluded.</p> <p><strong>Backgrounds inside Rules do not work</strong></p> <p>There's no sensible way to map Background up into the Feature level and only have it apply to some scenarios, therefore we throw a parser error if it happens.</p> <p>This syntax is discouraged by Cucumber anyhow</p>
<a href="https://github.com/Behat/Gherkin">Behat/Gherkin</a>
