[cucumber/godog] New release Release - published by vearutop

<https://github.com/cucumber/godog/releases/tag/v0.12.0|Release - >
<p>We are excited to announce the release of godog v0.12.0.</p> <p>Here follows a summary of Notable Changes, the Non Backward Compatible Changes and Deprecation Notices. The full change<br /> log is available <a href="https://github.com/cucumber/godog/blob/master/CHANGELOG.md">here</a>.</p> <p><strong>Notable Changes</strong> <strong>Output with multiple formatters</strong></p> <p>Now <code>godog</code> is able to use multiple formatters simultaneously with comma-separated formatters.</p> <p><code>--format pretty,junit:report.xml,cucumber:report.json</code> will write <code>pretty</code> format to stdout, <code>junit</code> to report.xml<br /> and <code>cucumber</code> to report.json.</p> <p><strong>Extensible formatters</strong></p> <p>Standard formatters are now exported with type aliases so that a custom formatter can be built on top of it.<br /> Please check &lt;/cucumber/godog/blob/_examples/custom-formatter|an example&gt;.</p> <p><strong>Contextualized hooks</strong></p> <p>Scenario and Step hooks are now passing context to allow custom state communication. Returned context should generally<br /> be based or equal to received context. Context is also passed to steps that have it in declaration and is read from<br /> steps that return it.</p> <p>Hooks can now return error, if non nil error is returned test is failed. This enables additional flow control, for<br /> example to check expectations after the scenario.</p> <p>Scenario hooks are now named <code>Before</code> and <code>After</code>.</p> <pre><code>// BeforeScenarioHook defines a hook before scenario. type BeforeScenarioHook func (ctx context.Context, sc **Scenario) (context.Context, error) // AfterScenarioHook defines a hook after scenario. type AfterScenarioHook func (ctx context.Context, sc **Scenario, err error) (context.Context, error) </code></pre> <p>Step hooks are now also named <code>Before</code> and <code>After</code>, but they are available with <code>ScenarioContext.StepContext()</code>.</p> <pre><code>// BeforeStepHook defines a hook before step. type BeforeStepHook func (ctx context.Context, st **Step) (context.Context, error) // AfterStepHook defines a hook after step. type AfterStepHook func (ctx context.Context, st **Step, status StepResultStatus, err error) (context.Context, error) </code></pre> <p><strong>Step definition improvements</strong></p> <p>Now <code>godog</code> can use additional ways to declare step definition. These declarations are optional and do not break<br /> backwards compatibility.</p> <p>Error result may be omitted if the step does not fail.</p> <pre><code>func iEat(arg1 int) { // Eat arg1. } </code></pre> <p>You can have <code>context.Context</code> as first argument, test runner will pass current context to the step.</p> <pre><code>func iEat(ctx context.Context, arg1 int) { if v, ok := ctx.Value(eatKey{}).int; ok { // Eat v from context. } // Eat arg1. } </code></pre> <p>You can have <code>context.Context</code> in return, test runner will use returned context to pass to next hooks and steps.</p> <pre><code>func iEat(ctx context.Context, arg1 int) context.Context { if v, ok := ctx.Value(eatKey{}).int; ok { // Eat v from context. } // Eat arg1. return context.WithValue(ctx, eatKey{}, 0) } </code></pre> <p>If error is also needed in return, context have to be first.</p> <pre><code>func iEat(ctx context.Context, arg1 int) (context.Context, error) { if v, ok := ctx.Value(eatKey{}).int; ok { // Eat v from context. } // Eat arg1. if arg1 == 0 { return errors.New("can't eat nothing") } return context.WithValue(ctx, eatKey{}, 0), nil } </code></pre> <p>You can
