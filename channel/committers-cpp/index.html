muggenhor (https://github.com/muggenhor)
#1183 Wire protocol broke backwards compatibility for docstrings
<p><strong>Summary</strong></p> <p>Starting from no later than cucumber 2.0 backwards compatibility of the wireprotocol got broken for passing docstrings.</p> <p>Given this step definition (cucumber-cpp):</p> <pre><code>GIVEN("some step with a docstring") { REGEX_PARAM(std::string, docstring); EXPECT_EQ("doc string goes here", docstring); } </code></pre> <p>and this feature:</p> <pre><code>Feature: test Scenario: test Given some step with a docstring """ doc string goes here """ When not implemented Then not implemented </code></pre> <p><strong>Expected Behavior</strong></p> <p>I expect the step definition to be called and passed the doc string. This requires this bit of JSON to fly across the wire (the <code>id</code> field is an example):</p> <pre><code>["invoke", { "id":"27", "args":[ "doc string goes here" ] } ] </code></pre> <p><strong>Current Behavior</strong></p> <p>With cucumber 1.3.8 (delivered with Ubuntu Trusty 14.04) the expectation is met, with Cucumber 2.0.0 (delivered with Ubuntu Xenial 16.04) however this fails. Instead an exception is thrown, from the implementation of the <code>REGEX_PARAM</code> macro, carrying the message "Parameter not found". This is the result of the transmitted JSON not being usable anymore, because it now looks like this instead:</p> <pre><code>["invoke", { "id":"27", "args":[ { "delegate_sd_obj":{ "content":"doc string goes here", "content_type":"", "location":"features/some-file.feature:25..27", "delegate_sd_obj":"doc string goes here" } } ] } ] </code></pre> <p><strong>Possible Solution</strong></p> <p>This seems to be the cause of serializing <a href="https://docs.ruby-lang.org/en/2.4.0/Delegator.html">Delegator</a> structures internally.</p> <p><strong>Steps to Reproduce (for bugs)</strong></p> <p>Running the FeatureShowcase example on the <a href="https://github.com/muggenhor/cucumber-cpp/commits/docstring-wire-bug">docstring-wire-bug branch of muggenhor/cucumber-cpp</a> with cucumber 1.3.8 will succeed, with 2.0 it will fail.</p> <p><strong>Context &amp; Motivation</strong></p> <p>This keeps my company from being able to upgrade our CI infrastructure to more recent Ubuntu versions as it ships with Cucumber 2.0.</p> <p><strong>Your Environment</strong></p> <p>Ubuntu Xenial 16.04 with Cucumber 2.0, cucumber-cpp HEAD.</p>

Rich picture
Part of the soft systems methodology, rich pictures provide a mechanism for learning about complex or ill-defined problems by drawing detailed ("rich") representations of them. Typically, rich pictures follow no commonly agreed syntax, usually consist of symbols, sketches or "doodles" and can contain as much (pictorial) information as is deemed necessary. The finished picture may be of value to other stakeholders of the problem being described since it is likely to capture many different facets of the situation, but the real value of this technique is the way it forces the creator to think more deeply about the problem and understand it well enough to express it pictorially (a process known as action learning). Rich pictures are a diagrammatic way of relating your own experiences and perceptions to a given problem situation through the identification and linking of a series of concepts. The creation of a rich picture provides a forum in which to think about a given situation. Rich pictures should concentrate on both the structure and the processes of a given situation. Rich pictures are a part of the understanding process, not just a way of recording what you know of a given situation or creating a work of art. The use of metaphor in rich pictures means that their interpretation by others can often be difficult. This is of little consequence as it is the personal learning aspects that are important to this method. It is also possible to produce rich pictures as part of a group. By having everybody contribute to a rich picture they can be used to help develop a shared understanding of a given situation. Mind maps are often considered to be rich pictures, but since these tend to be mainly text-based and do dictate a degree of formality with respect to their structure, clear distinctions can be drawn. Generally, the two notations serve different purposes (for example, in the field of systems engineering).
