*Thread Reply:* Hi Blaise,

BDD is a way to complement an Agile process, by having early conversations involving all parties: business, developers, testers, designers, etc. This conversation helps discover unknown unknowns and clarify intended behaviors.

Once they have a shared understanding, they formulate them into scenarios. These tests will serve as guide rails, to ensure that what is delivered matches what have been understood collectively. Ideally this should be readable by lesser technical people so that anyone can review them and give feedback.

These scenarios are then automated and used as a driver to implement the intended behaviors.

Using BDD does not mean using Cucumber, or Gherkin. Any tool fitting the domain problem and being understood by all parties can do.

In the BitBar blogpost, if I understand correctly they used the Swift protocol to describe the behavior they intended to implement, like a contract. They then automated by adding a layer of functions starting with given/when/then to make it more intention-revealing.

It looks like a valid Behaviour-Driven Development approach to me. There was no swift tool available that fitted their need so they developed one.

I dropped out from here when it was decided by higher management to revert QA to the glory 90:s in the company I work for. I left QA and went into DevOps instead. However, I wanted to post what I thought is very valuable insight. (I don't know how much people here follow John Ferguson Smart. He has a Youtube video very much the same as this text.)

Why do we want to automate tests in agile?

Test automation started out when testers wanted to make it more efficient to do the testing that had to be done to verify functionality after the production code had been written. The waterfall way, it was done after the development phase. Production code design was completed and it was necessary to verify that it did what product management had committed to when it had been sold. This verification phase would also take a number of weeks, with feedback to developers when defects were found and then handling new versions to repeat testing on. It was quite complicated and took code version management just to do testing.

But now we use agile. We don't wait for all code to be completed.

So what is it that we want to automate? Is it the testing of production code that was created after a sprint? But isn't that pretty much the same as before agile? The only difference is that the time is much more limited (within the sprint) and that the code base to test is (hopefully) a lot smaller. But it's still the same in principle. It's still just as much the waterfall process as it was back in the day.

So what testing is it we really want to automate? Or rather, why do we want to automate testing?

Why is much easier to answer! Because we want to be fast. Fast about detecting defects and fast about providing feedback to the developers so that the code that is created is as free from flaws as possible when we deliver.

This is the real value of automation! Fast feedback!

The real value of of automation is not to script manual tests at the end of a development cycle since that causes much delayed feedback to developers about functionality that already has been worked on several weeks earlier, and complicated version management of code to test. Even though there is a value of running automatic regression tests, the value of new automation is not to focus on creating regression tests for "old" functionality.

What really we want is to get feedback on the accuracy of the new code as soon as possible. So the automation that we do must help us achieve that. This is why we want to automate tests when working agile.
